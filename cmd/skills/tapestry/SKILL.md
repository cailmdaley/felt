## Why

Science moves fast. Computations proliferate, claims multiply, and the connection between them blurs. The tapestry makes the process inspectable and steerable: every computation is a node with evidence, every claim has provenance, the whole DAG is navigable. An agent leaves a legible wake not by slowing down, but by filing as it goes.

## When to Update the Tapestry

Not every fiber belongs — only what's important to inspect: computations with results, claims that need grounding, decisions that shaped the analysis.

- **After a computation produces results** → ensure the fiber has a `tapestry:` tag, write or update evidence.json
- **After finding a result** → update the fiber outcome with the finding, cite the exact source line
- **After making a decision** → file a decision fiber with alternatives considered
- **When told to "update the tapestry"** → check which computations ran, which fibers are missing tags or evidence, wire any new dependencies

## Format Conventions

These are the structural conventions that `felt tapestry export` implements. They are the single source of truth.

### Tapestry tags

`tapestry:{specName}` routes a fiber into the tapestry DAG. The suffix is the **spec name** — it links the fiber to its evidence directory. One fiber per specName.

```bash
felt add "B-modes consistent with noise" -t tapestry:bmodes
```

Only edges where **both endpoints** have `tapestry:` tags render in the DAG. Dependencies to non-tagged fibers exist in felt but don't appear as graph edges.

### Evidence directory

Evidence lives at `results/tapestry/{specName}/evidence.json`. The specName must match exactly in three places:

1. The `tapestry:` tag suffix
2. The directory name under `results/tapestry/`
3. The snakemake rule name (if evidence is generated by a pipeline)

### Evidence JSON schema

```json
{
  "evidence": {
    "pte": 0.29,
    "chi2": 12.3,
    "dof": 10
  },
  "output": {
    "figure": "bmodes.png",
    "table": "results.csv"
  },
  "generated": "2025-12-09T00:09:04Z"
}
```

| Field | Purpose |
|-------|---------|
| `evidence` | Flat dict of key metrics — numbers and short strings only. No nesting. |
| `output` | Named outputs — **only image files listed here render as artifacts** (`.png`, `.jpg`, `.jpeg`, `.pdf`). Non-image outputs are preserved in metrics but not copied as artifacts. |
| `generated` | ISO timestamp of when the evidence was produced. |

The file's **mtime** is used for staleness computation (not the `generated` field).

**What makes good evidence**: metrics that resolve the fiber's question (PTE, chi2, dof), key parameters that contextualize them (bin count, scale range, n_sims), diagnostic quantities (convergence flags, sample size). Numbers and short strings only — judgment and narrative go in the fiber outcome. The evidence is regenerable; never edit by hand.

### Staleness

For each tapestry node with evidence, `felt tapestry export` compares its evidence.json mtime against all upstream dependencies' evidence mtimes:
- Any upstream newer → **stale**
- No evidence.json → **no-evidence**
- Otherwise → **fresh**

### Sections (tier:1)

Sections correspond to the main parts of a paper (Introduction, Data, Methods, Results, Discussion). They are waypoints, not buckets — they link to each other in the DAG, and interior fibers belong to a section if they're 1 hop from it.

The `tier:1` tag makes a fiber a section: visible on first load, slightly larger, column-anchored in layout.

```bash
# Section node (visible on first load)
felt add "Methods" -t tapestry:methods -t tier:1

# Interior node
felt add "Covariance estimation" -t tapestry:covariance
felt link covariance-id methods-id
```

Section bodies should be short summaries of their neighborhood (3 sentences: what's here, why it matters, what a reader finds).

## Recording a Computation

Every computation that matters should have two things:

1. A **fiber** — the claim, context, and question the computation answers
2. An **evidence.json** — the machine-readable record of what ran and what it found

```bash
# Create the fiber
felt add "B-modes consistent with noise" -t tapestry:bmodes
felt edit bmodes-id -o "PTE=0.29, chi2=12.3/10 dof — consistent with noise at all scales"

# Wire to upstream (both endpoints must be tapestry-tagged)
felt link bmodes-id covariance-id
```

## Citing Sources

Paper sources go in `results/references/{arxiv-id}.tex`. In fiber outcomes, cite as inline code:

```
`results/references/2601.10038.tex:L79`
```

Accepts `:L42` (single line) and `:L42-55` (range). Any inline code matching `path/to/file:L42` becomes a clickable link in the static viewer.

## Exporting

```bash
felt tapestry export              # writes to ~/.felt/tapestries/data/{project}/
felt tapestry export --all-fibers # include all fibers (for sidebar)
felt tapestry export --force      # re-copy all artifacts
felt tapestry export --name foo   # override city name
```

Output goes to `~/.felt/tapestries/`, which should be a clone of a tapestry template repo (GitHub Pages). The export writes `tapestry.json` + artifact images. Git commit/push is a separate step.

## Phases of Growth

A tapestry grows through phases, and you can't skip them:

1. **Inspectable** — every computation has evidence, every node has a fiber, the DAG is wired. This is the diagnostic foundation. It's necessary but not sufficient — 30 nodes with 175 plots is inspectable but not comprehensible.

2. **Comprehensible** — the tapestry reads as a story. Nodes are regions of a map, not items in an inventory. A reader enters on the left, moves through naturally, and emerges understanding the state of the analysis. Per-wildcard plots compress into composites. Infrastructure nodes recede; what matters is what the computation *reveals*.

3. **Living** — the tapestry continues to grow. New analyses add nodes; shaping keeps the narrative coherent.

A constitution (see `/constitution`) can drive each phase.
