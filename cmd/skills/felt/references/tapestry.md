# Tapestry

The tapestry is the human-facing surface of the work. It transforms the chaos of fibers — decisions, actions, bugs, emails, technical fixes — into something polished and navigable. Not every fiber belongs. Only what helps a human understand the state of things.

For scientific analysis: computations with evidence, key claims, the shape of what's known and unknown.
For projects: documents, research, landscape context, institutional facts, key open questions.

**NOT:** decisions, actions, emails, formatting fixes, task tracking, technical migrations. These live in the DAG for the agent to trace. The tapestry is what the human reads. Aim for ~10–15 nodes — enough to orient, few enough to hold in your head.

## When to Update

- **After a computation produces results** → ensure the fiber has a `tapestry:` tag, write or update evidence.json
- **After finding a result** → update the fiber outcome with the finding, cite the exact source line
- **When told to "update the tapestry"** → check which fibers are missing tags or evidence, wire any new dependencies

## Branching Discipline

Tapestry nodes follow a strict branching budget for progressive disclosure:

- **2–4 children** per node is healthy
- **3** is the default target
- **5+** triggers reshaping — introduce a grouping node

**Grouping nodes** are not empty containers. They carry a summary of their neighborhood (3 sentences: what's here, why it matters, what a reader finds). They don't need evidence, but they must convey information. A reader should understand the group's scope without clicking into it.

When filing a new tapestry fiber:
1. Attach to the most specific existing node whose question this fiber resolves
2. If that parent would exceed 4 children, look for an existing subgroup
3. If none exists, create a grouping node named by the natural partition that just emerged

**Reshaping** is a periodic pass (not every filing). See [maintenance.md](maintenance.md#tapestry-reshaping) for the full procedure.

## Format Conventions

These are the structural conventions that `felt export --format tapestry` implements.

### Tapestry tags

`tapestry:{specName}` routes a fiber into the tapestry DAG. The suffix is the **spec name** — it links the fiber to its evidence directory. One fiber per specName.

```bash
felt add "B-modes consistent with noise" -t tapestry:bmodes
```

Only edges where **both endpoints** have `tapestry:` tags render in the DAG. Data-flow edges to non-tagged fibers can still exist via `inputs.from`, but they do not appear as graph edges.

### Evidence directory

Evidence lives at `results/tapestry/{specName}/evidence.json`. The specName must match exactly in three places:

1. The `tapestry:` tag suffix
2. The directory name under `results/tapestry/`
3. The snakemake rule name (if evidence is generated by a pipeline)

### Evidence JSON schema

```json
{
  "evidence": {
    "pte": 0.29,
    "chi2": 12.3,
    "dof": 10
  },
  "output": {
    "figure": "bmodes.png",
    "table": "results.csv"
  },
  "generated": "2025-12-09T00:09:04Z"
}
```

| Field | Purpose |
|-------|---------|
| `evidence` | Flat dict of key metrics — numbers and short strings only. No nesting. |
| `output` | Named outputs — **only image files listed here render as artifacts** (`.png`, `.jpg`, `.jpeg`, `.pdf`). Non-image outputs are preserved in metrics but not copied as artifacts. |
| `generated` | ISO timestamp of when the evidence was produced. |

The file's **mtime** is used for staleness computation (not the `generated` field).

**What makes good evidence**: metrics that resolve the fiber's question (PTE, chi2, dof), key parameters that contextualize them (bin count, scale range, n_sims), diagnostic quantities (convergence flags, sample size). Numbers and short strings only — judgment and narrative go in the fiber outcome. The evidence is regenerable; never edit by hand.

### Staleness

For each tapestry node with evidence, `felt export --format tapestry` compares its evidence.json mtime against all upstream dependencies' evidence mtimes:
- Any upstream newer → **stale**
- No evidence.json → **no-evidence**
- Otherwise → **fresh**

Staleness propagates transitively through nodes without evidence (grouping nodes). A grouping node never blocks staleness — the check walks through it to find the nearest upstream evidence.

### Spine nodes (tier:1)

The spine is the set of high-level nodes that give a tapestry its orientation. For a research paper these might be Data, Methods, Results; for a software project they might be Architecture, API, Deployment. The spine is whatever makes a newcomer say "I see what this is about" before clicking anything.

The `tier:1` tag marks a spine node: visible on first load, slightly larger, column-anchored in layout. Interior fibers belong to a spine node if they're 1 hop from it.

```bash
# Spine node (visible on first load)
felt add methods "Methods" -t tapestry:methods -t tier:1

# Interior node
felt add covariance-estimation "Covariance estimation" -t tapestry:covariance
felt nest covariance-estimation methods
```

## Recording a Computation

Every computation that matters should have two things:

1. A **fiber** — the claim, context, and question the computation answers
2. An **evidence.json** — the machine-readable record of what ran and what it found

```bash
# Create the fiber
felt add b-modes-consistent-with-noise "B-modes consistent with noise" -t tapestry:bmodes
felt edit b-modes-consistent-with-noise -o "PTE=0.29, chi2=12.3/10 dof — consistent with noise at all scales"

# Place it under the relevant spine node
felt nest b-modes-consistent-with-noise covariance-estimation
```

## Citing Sources

Paper sources go in `results/references/{arxiv-id}.tex`. In fiber outcomes, cite as inline code:

```
`results/references/2601.10038.tex:L79`
```

Accepts `:L42` (single line) and `:L42-55` (range). Any inline code matching `path/to/file:L42` becomes a clickable link in the static viewer.

## Exporting

```bash
felt export --format tapestry              # writes to ~/.felt/tapestries/data/{project}/
felt export --format tapestry --all-fibers # include all fibers (for sidebar)
felt export --format tapestry --force      # re-copy all artifacts
felt export --format tapestry --name foo   # override city name
```

Output goes to `~/.felt/tapestries/`, which should be a clone of a tapestry template repo (GitHub Pages). The export writes `tapestry.json` + artifact images. Git commit/push is a separate step.

## Phases of Growth

A tapestry grows through phases, and you can't skip them:

1. **Inspectable** — every computation has evidence, every node has a fiber, the DAG is wired. This is the diagnostic foundation. It's necessary but not sufficient — 30 nodes with 175 plots is inspectable but not comprehensible.

2. **Comprehensible** — the tapestry reads as a story. Nodes are regions of a map, not items in an inventory. A reader enters on the left, moves through naturally, and emerges understanding the state of the analysis. Per-wildcard plots compress into composites. Infrastructure nodes recede; what matters is what the computation *reveals*.

3. **Living** — the tapestry continues to grow. New analyses add nodes; shaping keeps the narrative coherent.

A constitution (see [constitute.md](constitute.md)) can drive each phase.
